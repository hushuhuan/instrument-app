Per `.specify/memory/constitution.md` (Think Before Coding, Simplicity First, Surgical Changes, Goal-Driven Execution, Python stack, and related principles):

- **Python stack (Principle IX)**: Application implementation is **Python** unless this plan documents a constitution-approved exception (subsystem name + rationale). **Language/Version** and dependencies below MUST reflect that choice (e.g. PySide6/PyQt6 for Qt-style desktop apps).
- **Assumptions**: Material assumptions for this plan are stated explicitly; gaps are flagged (e.g. `NEEDS CLARIFICATION`), not silently guessed.
- **Ambiguity**: If the spec allows multiple interpretations, those options are listed with tradeoffs—no silent choice of one reading.
- **Simplicity & pushback**: A simpler approach is named when it exists; if a more complex design is chosen, the plan states why the simpler path was rejected or insufficient.
- **Simplicity First (implementation)**: Planned work stays within spec scope—no speculative features, layers, configurability, or extensibility unless traced to requirements.
- **Surgical Changes**: Implementation tasks derived from this plan target only files and behaviors needed for the spec; drive-by refactors or unrelated cleanup require an explicit task or follow-up, not silent inclusion.
- **Goal-Driven Execution**: Tasks and milestones from this plan state how each will be verified (tests, commands, or measurable outcomes)—not vague "complete" labels; multi-step work lists `step → verify` checkpoints.
- **Stop rule**: If requirements or constraints remain confusing, planning pauses until clarification is recorded (e.g. via `/speckit-clarify` or spec update).
- **Traceability**: Plan decisions map to the feature spec; any deviation is documented with rationale.
