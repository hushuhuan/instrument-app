# Feature Specification: 有机合成仪器控制软件（流程与 GUI）

**Feature Branch**: `001-organic-synthesis-control`  
**Created**: 2026-05-05  
**Status**: Draft  
**Input**: User description: 有机合成仪器控制软件，关注业务流程与 GUI；**业务运行为六类可重复、顺序可编排的阶段**（加粉、反应、萃取、蒸馏、过滤、取样）；GUI 含反应参数与工艺设定；底层支持注射泵、旋转阀、电磁阀、电磁铁、电机等。

## Clarifications

### Session 2026-05-05

- Q: GUI 是否需状态栏及展示内容？ → A: 需要全局状态栏；持续显示温湿度、当前任务（步骤）名称、当前流程进度。
- Q: 状态栏刷新与温湿度/进度展示细则？ → A: 刷新周期 **1 s**；环境温度可切换 **℃（默认）/℉**，软件内换算与切换；湿度以 **%RH** 显示；进度展示 **整体剩余时间** + **当前执行动作**；当前动作与 **`businessPhase`** 同名（六类阶段）。
- Q: 业务运行流程如何编排？ → A: 运行由 **加粉、反应、萃取、蒸馏、过滤、取样** 六类**业务阶段**组成；方法中为**有序步骤列表**，每一步对应其一；**同一阶段可重复出现多次**；**执行顺序可完全由操作员在方法中调整**（无固定模板顺序）。底层设备动作仍由各阶段内部脚本映射（见 plan）。
- Q: 是否需要仪器自检？ → A: 需要；主界面提供 **「自检」按钮**；自检依次（顺序见 plan）包含：**网络检查**、**磁盘剩余容量检查**、**注射泵初始化**、**旋转阀初始化**、**步进电机回原点**、**冷热水机通讯**；每步有明确通过/失败与汇总报告。
- Q: 是否需要无硬件开发与演示能力？ → A: 需要；提供 **Mock（仿真）模式**，在该模式下 **HAL 层全部由软件模拟实现**，可运行完整 GUI、自检（设备子项为模拟）、业务流程与状态栏，无需真实仪器（见 FR-018 与 plan）。
- Q: 用户能否只提交「反应任务」并执行？ → A: 能；提供 **反应任务** 提交与执行路径（生成至少含 **`反应`** 阶段的 Recipe 并启动 Run）；**Mock 模式**下须可完整跑通（见 **User Story 5**、FR-019）。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 在 GUI 中设定并校验反应与工艺参数 (Priority: P1)

实验室操作员在一次实验开始前，通过界面输入或选择本机实验所需参数：反应时间、搅拌速度、反应温度、冲洗参数、溶剂、萃取参数、过滤参数、取样数量等；系统在提交或进入下一步前校验参数是否在允许范围内，并对明显不合规组合给出提示。

**Why this priority**: 无可用参数表则后续自动化流程无法安全启动，是整体产品的最小可用价值。

**Independent Test**: 在不连接真实硬件或使用硬件仿真/桩的情况下，仅通过界面与校验规则即可验证：必填项、上下限、单位一致性及提示文案是否满足要求。

**Acceptance Scenarios**:

1. **Given** 操作员打开参数设定界面，**When** 输入在允许范围内的各项参数并确认，**Then** 系统接受参数并可供后续流程引用且无错误提示。
2. **Given** 操作员输入超出上下限的温度或时间为非法值，**When** 尝试保存或进入运行准备，**Then** 系统阻止继续并指出具体字段与原因。
3. **Given** 某溶剂或冲洗/萃取模板尚未选择，**When** 操作员跳过必填项，**Then** 系统明确提示缺失项，不允许进入“可启动流程”的就绪状态。

---

### User Story 2 - 编排并执行业务阶段序列（可重复、顺序可调）(Priority: P2)

操作员在实验方法中定义一条**业务运行序列**：每一步必须是 **加粉、反应、萃取、蒸馏、过滤、取样** 六类阶段之一；**同一阶段可在序列中出现多次**；**步骤先后次序由操作员自由编排**（无强制固定流程模板）。运行中系统按列表顺序执行，界面展示当前阶段、已完成/失败阶段及整体进度；在允许的情况下可暂停、恢复或安全中止。

**Why this priority**: 核心价值是可重复的机控流程，且贴合真实实验中“反复萃取/多次反应”等需求。

**Independent Test**: 使用 Mock HAL，构造至少含 **两处相同阶段相邻出现**、且顺序**非字典序**的方法，验证执行次序与列表一致；状态栏当前动作与列表当前项一致。

**Acceptance Scenarios**:

1. **Given** 方法中步骤顺序为「萃取 → 反应 → 萃取」，**When** 自起点运行，**Then** 系统先执行萃取再反应再萃取，且状态栏与主界面同步显示当前阶段名称。
2. **Given** 方法含 3 个「反应」步骤分散在不同位置，**When** 完整运行，**Then** 三者均被依次执行且可分别报告成功/失败。
3. **Given** 运行中某阶段失败，**When** 故障被检测，**Then** 系统进入告警/失败态，阻止未经确认的继续执行后续步骤（除非方法或操作规程明确允许跳过——本版默认不允许自动跳过）。
4. **Given** 流程处于运行或暂停状态，**When** 操作员注视主窗口底部状态栏，**Then** 可见当前阶段名称、与主流程区一致的剩余时间与进度语义，以及温湿度或无效提示（FR-011、FR-012）。

---

### User Story 5 - 提交并执行反应任务（可 Mock）(Priority: P2)

操作员通过专用入口填写/确认一次性 **反应任务**（至少包含反应时间与反应温度；其他 FR-001 字段可选用或系统缺省），点击 **提交** 后，系统生成 **Recipe**（`steps[]` **至少一行** `businessPhase=反应`，`planDurationSec` 与反应时间一致）并 **自动启动 Run**。在 **Mock 模式**（FR-018）下，全过程 **无需物理硬件** 即可推进至 `Completed` 或 `Failed`，状态栏与主流程区与普通方法运行一致。

**Why this priority**: 支持最常见的单步反应验证、培训与联调，独立于完整多阶段方法编排仍有独立交付价值。

**Independent Test**: 仅启动 `--mock`，自打开反应任务对话框至 Run 结束，无需使用完整方法编辑器。

**Acceptance Scenarios**:

1. **Given** 应用在 Mock 模式且会话空闲，**When** 操作员输入合法反应时间与温度并提交反应任务，**Then** Run 启动且当前阶段显示 **「反应」**，结束后进入 `Completed` 或明确的 `Failed`。
2. **Given** 反应时间或温度非法，**When** 提交，**Then** 系统阻止启动并与 FR-002 一致的提示。
3. **Given** 已有 Run 为 Running/Paused 或自检占用会话，**When** 尝试提交反应任务，**Then** 系统阻止并提示需空闲（与全局互斥一致）。

---

### User Story 3 - 统一调度底层设备能力以支撑流程 (Priority: P3)

在执行各业务步骤时，系统自动将逻辑动作映射为对设备能力的调用：注射泵流体输送、旋转阀通道选择、电磁阀通断、电磁铁吸合/释放、电机运动（如搅拌或驱动相关机构）。操作员在常规界面不直接操作寄存器级细节，但能看到与安全和进度相关的汇总状态（如泵 busy、阀位、电机异常）。

**Why this priority**: 业务流程的可靠性取决于设备抽象是否一致、可观测、可失败处理。

**Independent Test**: 针对每类设备能力，在台架或仿真环境下验证：单步手动测试模式（若需求启用）可令单一动作执行并返回预期状态；在完整流程中验证组合动作无死锁、互斥资源使用正确。

**Acceptance Scenarios**:

1. **Given** 流程处于「反应」阶段且该步需要液体输送，**When** HAL 执行该阶段内部脚本，**Then** 注射泵按该步 `deviceArgs` 完成可追溯动作并上报完成或失败。
2. **Given** 流程需要在多条流路间切换，**When** 执行依赖流路选择的步骤，**Then** 旋转阀/电磁阀组合达到方法要求的通断状态，状态与步骤一致。
3. **Given** 某电机或电磁铁动作未在规定时间内完成，**When** 传感器或驱动层上报超时，**Then** 系统标记步骤失败并触发 User Story 2 中的告警行为。

---

### User Story 4 - 仪器自检（维护/实验前检查）(Priority: P2)

实验员或维护人员在开机后、实验开始前，点击 **「自检」** 启动一次性诊断流程。系统按既定顺序执行：**网络**、**磁盘余量**、**注射泵初始化**、**旋转阀初始化**、**步进电机回原点**、**冷热水机通讯**（术语与具体判定规则以计划与配置为准）。界面逐步显示进行中项与最终结果；任一步失败时后续未执行项可标为 **跳过** 或 **未执行**（见 FR-016），并给出可读的失败原因；全部通过后形成可复查的汇总状态。

**Why this priority**: 降低带故障运行风险，与设备初始化直接相关，独立于具体合成方法但仍为日常操作高频路径。

**Independent Test**: 使用 Mock HAL + 模拟磁盘/网络桩，可注入单步失败，验证 UI 报告与顺序；无需真实化学品。

**Acceptance Scenarios**:

1. **Given** 主界面可用，**When** 操作员点击 **「自检」**，**Then** 自检开始且首项为网络相关检查（与计划一致）。
2. **Given** 磁盘剩余空间低于配置的最低阈值，**When** 执行磁盘检查步骤，**Then** 该步标记为失败并展示原因，且操作员可见汇总为失败。
3. **Given** 泵初始化成功而旋转阀初始化失败，**When** 流程结束，**Then** 汇总中泵为通过、旋转阀为失败，后续依赖项行为符合 FR-016。
4. **Given** 全流程成功，**When** 自检结束，**Then** 汇总显示六项均为通过（或配置为可选项的项显示「跳过/不适用」时须在汇总中区分），且可再次启动自检。

### Edge Cases

- 状态栏所示 **环境温湿度** 与 **反应工艺设定/釜内控制温度**（FR-001）可能不同源；界面 MUST 区分语义，避免操作员将二者混读（标签或分区在计划中细化）。
- 与设备的通信中断或部分通道失效时，系统如何提示并是否允许降级为纯手动旁路（假设：默认禁止未确认的自动继续，需人工确认）。
- 反应放热等导致温度超限：温控与报警联动，是否自动停止加热/反应（假设：达到危险阈值必须自动进入安全停机）。
- 并行请求冲突（如搅拌与某阀动作在同一互斥窗口）：系统必须序列化或拒绝非法组合并提示。
- 操作员在流程中途修改参数：区分“未运行可改”与“运行中锁定”；运行中变更需明确定义是否允许及如何生效。
- 操作员自定义阶段顺序时，**化学/工艺合理性**由实验方案负责人保证；v1 软件 **不强制** 特定的阶段先后合法性规则（除安全互锁/设备能力约束外）。若未来需“禁止某两类相邻”，应单列需求版本。
- **自检与业务运行并行**：业务方法 **Running** 时 MUST 禁止启动自检，或 MUST 提示先暂停/中止并获确认（具体以 FR-016 为准）；默认 **禁止并行** 以避免设备命令冲突。

## Requirements *(mandatory)*

### Functional Requirements

**GUI — 反应与工艺参数设定**

- **FR-001**: 系统 MUST 提供界面用于设定反应时间、搅拌速度、反应温度、冲洗参数、溶剂选择、萃取参数、过滤参数与取样数量。
- **FR-002**: 系统 MUST 对 FR-001 中各字段提供可配置的合法范围与必填规则，并在违反时阻止进入流程就绪态。
- **FR-003**: 系统 MUST 以操作员可理解的方式展示当前参数集摘要，便于实验前核对。

**GUI — 全局状态栏**

- **FR-011**: 系统 MUST 在主窗口提供常驻、全局可见的状态栏，并按 **不高于 1 秒** 的周期刷新显示：（1）环境温度、**相对湿度（%RH）**；环境温度 MUST 支持操作员在 **℃（默认）** 与 **℉** 之间切换，显示值由软件按同一传感器读数换算；（2）**当前正在执行的业务阶段名称**（与当前 `RecipeStep` 的 `businessPhase` 一致，六类之一；见 FR-004）；（3）**整体剩余时间估计** 及与 FR-005 一致的进度语义。流程未启动时显示空闲或等价文案，暂停时对剩余时间及阶段展示的行为在计划中约定（须避免误导）。
- **FR-012**: 当温湿度数据源不可用、超时或读数明显无效时，系统 MUST 在状态栏明示数据不可用或陈旧，而 MUST NOT 静默显示可能误导的数值。

**业务流程**

- **FR-004**: 系统 MUST 将一次实验运行的业务逻辑表示为**有序步骤列表** `steps[]`；列表中每一步的 `businessPhase` MUST 为以下之一：**加粉、反应、萃取、蒸馏、过滤、取样**。**同一 `businessPhase` 允许在列表中出现多次**；**步骤顺序完全由方法定义决定**，系统 MUST **不提供**隐含的固定阶段顺序模板（可提供空白方法或示例文件，但不强制）。运行 MUST **严格按 `steps[]` 下标递增**执行。
- **FR-013**: 系统 MUST 提供方法编辑能力，使操作员能够对 `steps[]` 执行：**添加**步骤（选定阶段类型）、**删除**步骤、**调整顺序**（任意重排）、**复制**步骤（可选快捷，便于重复配置同类阶段）；保存后的列表即为唯一运行次序。
- **FR-005**: 系统 MUST 在界面展示当前步骤索引、各步状态（未开始/进行中/成功/失败）及整体进度；**当前阶段显示名** MUST 与状态栏（FR-011）一致。
- **FR-006**: 系统 MUST 在安全条件允许时支持暂停、恢复与安全中止；中止后设备处于方法定义的安全待机或停机姿态（具体姿态由硬件方法表约定，软件须执行该方法表）。

**底层设备能力（逻辑需求，不限定协议）**

- **FR-007**: 系统 MUST 能够通过统一接口驱动或协调：注射泵、旋转阀、电磁阀、电磁铁、电机，以完成各 **`businessPhase`** 所需的内部子动作（子动作对操作员默认不可见，见 plan 中阶段脚本）；硬件互锁与安全约束 MUST 优先于用户编排顺序。
- **FR-008**: 系统 MUST 采集并展示与安全及排障相关的设备汇总状态（如通讯健康、忙闲、故障码的人类可读映射），禁止仅暴露不可解释的原始代码作为唯一信息。
- **FR-009**: 系统 MUST 在设备动作失败、超时或互锁触发时产生可记录的告警事件（时间、步骤、设备类别、结果）。

**数据与可追溯（研发场景假设）**

- **FR-010**: 系统 MUST 允许将一套完整参数与步骤序列保存为具名的实验方法，并再次加载；保存内容足以复现同序步骤与参数（不必包含商业化电子签名，见假设）。

**仪器自检**

- **FR-014**: 系统 MUST 在主界面或主导航区域提供**可见的「自检」入口**（按钮或等价控件），仅在空闲/安全状态下可启动（与 FR-016 一致）。
- **FR-015**: 自检 MUST 按固定顺序执行以下检查/动作：**（1）网络检查**、**（2）磁盘剩余容量检查**、**（3）注射泵初始化**、**（4）旋转阀初始化**、**（5）步进电机回原点**、**（6）冷热水机通讯**。顺序不可由操作员在 v1 中重排（与工艺阶段序列不同）；阈值与超时在配置中定义（计划中落地）。
- **FR-016**: 当业务运行处于 **Running** 或 **Paused**（即工艺 Run 未结束且非空闲态）时，系统 MUST **阻止**启动自检并提示原因；若某自检步骤失败，系统 MUST 中止后续自检步骤并保留已完成步骤的结果供查阅；失败原因 MUST 人类可读（FR-008 语义一致）。
- **FR-017**: 自检完成后，系统 MUST 向操作员展示**汇总结果**（每步：通过/失败/跳过及摘要文案）；结果 SHOULD 可写入日志以便事后追溯（与 FR-009 告警可区分存储）。

**仿真（Mock）运行**

- **FR-018**: 系统 MUST 提供 **Mock（仿真）模式**。在此模式下，经 HAL 访问的 **全部设备能力**（注射泵、旋转阀、电磁阀、电磁铁、步进/电机、冷热水机等）均由 **进程内 Mock 实现**替代，不得依赖真实外设即可：加载/编辑方法、执行业务阶段序列至完成或失败、执行自检（其中设备相关步骤走 Mock）、驱动状态栏环境读数（使用 **可配置的模拟传感器**轮廓）。Mock 模式 MUST 通过 **显式**启动开关激活（命令行参数、环境变量或配置文件字段，**唯一真相**见 plan）；主窗口 MUST **显著标示**当前处于仿真（例如标题栏或顶栏「仿真运行」），以免与真机混淆。

**反应任务**

- **FR-019**: 系统 MUST 提供 **反应任务** 功能：用户在界面提交反应任务后，系统 MUST 构建 **Recipe**（含 `parameters` 与 **仅含 `反应` 阶段或用户指定的以反应为主的步骤集**；v1 **最小**实现为 **单步** `businessPhase=反应`）并调用与 FR-005/006 相同的 **Run** 管线执行；**Mock 模式**下 MUST 可不接真机完成该管线至终态。

### Key Entities *(include if feature involves data)*

- **实验方法（Recipe / Method）**: 名称、版本或修订时间、参数集（FR-001）、**有序业务阶段列表** `steps[]`（FR-004；每步含 `businessPhase` 与 `planDurationSec` 等）及每步关键设备参数。
- **方法步骤（RecipeStep）**: 单一 `businessPhase`（六类之一）、在 `steps[]` 中的顺序位置、该步的 `planDurationSec` / `deviceArgs`、可选 `displayOverride`。
- **流程运行实例（Run）**: 关联方法、开始/结束时间、每步状态、告警列表。
- **设备能力描述（Capability）**: 逻辑名称（如“主注射泵”）、类别（泵/阀/电磁铁/电机）、允许的动作与状态枚举。
- **环境读数（AmbientReading）**: 与状态栏关联的温度（原始值 + 单位）、湿度（%RH）、采集时间戳、有效性标记；温度单位偏好（℃/℉）为用户/会话设置。
- **自检会话（SelfTestRun）**: 单次自检标识、开始/结束时间、各 **自检子项** 状态与消息、整体验证通过/失败。
- **运行模式（ApplicationRunMode）**: `Production` | `Mock`（FR-018）；决定 `InstrumentHal` 绑定真实适配器或 `MockInstrumentHal`。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 熟练操作员在单次会话中，可在 **15 分钟**内完成一份包含至少 **8 个参数项** 的典型实验方法录入、校验通过并保存为具名方法。
- **SC-002**: 对一份 **不少于 10 步** 且 **六类 `businessPhase` 各至少出现一次**、并 **至少含一类阶段重复出现 ≥2 次**、且 **非按固定模板排序** 的方法，在仿真或台架环境下 **至少 95%** 试运行可从中途无人工干预推进至“流程成功完成”或按设计预期进入“可恢复的失败停稳”状态（与注入故障用例区分）。
- **SC-003**: **100%** 的已知非法参数组合在进入“可启动”状态前被拦截，且在可用性测试中 **90%** 受试者能在 **1 次**提示内找到需修改的字段（定性通过实验室内部小规模测试记录）。
- **SC-004**: 发生注入的通信中断或单设备故障类用例时，系统能在 **5 秒内**进入告警与安全停稳路径，且界面呈现可归类原因（不要求区分底层总线类型）。
- **SC-005**: 在流程运行或暂停期间，状态栏中的 **当前任务名称** 与 **进度表示** 与主界面流程区在 **同一刷新节拍（默认 1 s，与 FR-011 一致）** 内一致；实验室走查抽样连续 **10 次** 观察无矛盾（含温湿度在有效/无效两种状态下状态栏行为符合 FR-012）。
- **SC-006**: 在 Mock 环境下，一次完整自检从点击到汇总呈现 **不超过 3 分钟**（各类硬件超时使用配置文件中的研发缺省值）；**连续 20 次** 全通过基准跑中 **至少 95%** 的汇总结论与注入的桩状态一致。单步失败注入时，**100%** 的案例在汇总中体现为失败且**不报告**未完成项为通过。
- **SC-007**: 在 **Mock 模式**（FR-018）下，使用与 **SC-002** 同结构复杂度的参考方法，**连续 5 次**「自开始运行 → Completed」无进程崩溃，且每次终态在流程视图与状态栏一致；若人为在 Mock 中注入单步失败，则 **100%** 呈现 Failed/安全停稳而非挂死。
- **SC-008**: 在 **Mock 模式**下，自 **「提交反应任务」（FR-019）** 至 Run 结束，**连续 5 次**无进程崩溃；运行期间状态栏当前阶段 **MUST** 为「反应」且 **剩余时间语义** 与主流程区一致（与 SC-005 同一节拍）。

## Assumptions

- 首版目标部署为 **受控实验室内单机单操作员** 使用；**制药 GMP 电子记录/电子签名、完整审计合规包** 不在本规格 v1 范围内，除非后续单独立项。
- **物理急停与安全联锁** 以硬件或固件为主；软件 **必须服从** 硬件互锁信号且不提供绕过互锁的隐性路径。
- **GUI 主语言为中文**；方法名、字段名可采用中英混合以便与国际供应商文档对照，但操作员可见文案以中文为主。
- 硬件通道数量、管路拓扑与标定系数以 **可配置数据** 形式提供，本规格不固定具体机型编号。
- “加粉”等步骤如涉及称重或视觉确认，首版假定 **已有外设反馈或可配置的手动确认点**；若现场无自动称重，则以操作员在界面确认“已加粉”作为门禁（实现时在计划中细化，不在本规格锁定交互样式）。
- 状态栏温湿度来自 **仪器或实验舱配置的环境传感器**（探头位置、精度与校准周期在计划中通过配置与运维约定）；无传感器或未配置时，状态栏按 FR-012 显示不可用。
- **网络检查**指与本机实验所需连通性相关的检查（例如默认网关/指定设备 IP 的可达性，**非**全面 IT 审计）；具体探测方式在计划中定义（如 `QNetworkInterface` + ping/TCP 探测占位）。
- **磁盘剩余容量**针对仪器数据/方法落盘分区，**最低剩余空间阈值**可配置（计划给出研发缺省，如 ≥ 1 GB）。
- **冷热水机**即工艺温控水路设备（冷水机/恒温循环等）的通讯握手成功判定，由 HAL 适配实现，不绑定单一厂商协议。
- **Mock 模式**下网络/磁盘检查仍可使用**真实 OS 信息**（真实网卡与磁盘），或按 plan 提供「全柔仿真」选项将二者也短路为 Pass（仅用于 CI）；默认 Mock 仍以真实磁盘/网络结果为准以便发现开发机环境问题。
